# Agent QA Acceleration Plan

**Created:** Feb 26, 2026
**Goal:** Replace manual walkthrough of voice-agent-test-scenarios.md and voice-agent-integration-test-scenarios.md with automated testing and continuous evaluation.

---

## Executive Summary

We have ~80+ manual test scenarios across 4 integration modes (No Integration, Google Calendar, Jobber, Housecall Pro) that currently require someone to walk through them by voice. This plan uses **two complementary tools** to automate QA:

1. **LiveKit pytest** — Automated behavioral tests that run in CI (pre-deployment gate)
2. **Langfuse** — Production observability, online evaluation, and offline experiments

These are not competing approaches — they serve different stages of the development loop:

| Stage | Tool | Purpose |
|-------|------|---------|
| Pre-deploy (CI) | LiveKit pytest | Catch regressions before they reach staging |
| Staging/Production | Langfuse tracing | Full visibility into every LLM/STT/TTS/tool call |
| Production monitoring | Langfuse online eval | Auto-score every live call for quality |
| Prompt iteration | Langfuse experiments | Compare prompt changes on datasets before deploy |
| Edge case capture | Langfuse → datasets | Turn real production traces into test cases |

## Part 1: Langfuse Prompt Management Evaluation

### Can we use Langfuse to manage our system prompts?

**Short answer: No, not for the main system prompt. The dynamic generation is too complex.**

Our system prompt is generated by `generateSystemPrompt()` in `src/utils.ts` — a ~5000-line TypeScript function that takes 30+ parameters and produces a highly conditional prompt. The prompt varies by:

- Integration type (none, GCal, Jobber, HCP, Square, Acuity)
- Returning vs new caller (skips already-known info)
- Intake questions configured per-location (dynamically numbered steps)
- Business hours, services, service areas, FAQ, discounts
- Caller addresses (single vs multi-address confirmation/selection)
- Field service vs scheduling workflow sections
- Safety instructions, address validation protocol, silence detection

Langfuse prompt management supports **variables** (`{{var}}`) and **composability** (`@@@langfusePrompt:name=X@@@`) but **no conditional logic** (`{% if %}` is not supported). Moving our prompt to Langfuse would require:

- ~50+ variables with application-level conditional selection of sub-prompts
- Net **increase** in complexity vs the current TypeScript function
- Loss of type safety and compile-time checks
- No benefit since prompt updates already deploy instantly (dynamic generation at call-start)

### What about sub-sections?

Some relatively static sections (safety instructions, address protocol, voice persona) *could* theoretically live in Langfuse, but the ROI is low — they're embedded in a dynamically generated prompt and would add latency + complexity for fetching.

### Recommendation

- **Keep** prompt generation in TypeScript (`src/utils.ts`)
- **Use** Langfuse for what it excels at: **tracing, evaluation, and experiments**
- **Link** traces to prompt versions by including a prompt hash/version in Langfuse metadata (free — just add to `setup_langfuse()` metadata)

## Part 2: Langfuse Evaluation & Experiments

### What's available

| Feature | Description | Value for us |
|---------|-------------|-------------|
| **Online LLM-as-a-Judge** | Auto-score production traces in real-time | HIGH — auto-evaluate every call for quality |
| **Annotation Queues** | Structured human review workflows | MEDIUM — review flagged/low-score calls |
| **Datasets** | Collections of test cases (input + expected output) | HIGH — automate test scenario regression |
| **Experiments via SDK** | Run task against dataset, compare results | HIGH — test prompt changes before deploy |
| **Custom Dashboards** | Visualize scores, latency, cost over time | HIGH — monitor quality trends |
| **Score Analytics** | Analyze evaluation scores | MEDIUM — identify systematic issues |

### Online Evaluation (Production Monitoring)

Set up LLM-as-a-Judge evaluators that automatically run on every production trace:

| Evaluator | What it checks | Score type |
|-----------|---------------|------------|
| **Tool Usage Correctness** | Did the agent call the right tools in the right order? | 0-1 numeric |
| **Safety Compliance** | Did the agent handle emergencies correctly (gas leak, flooding)? | boolean |
| **Intake Completeness** | Did the agent collect name + address before booking? | boolean |
| **Tone & Professionalism** | Was the agent professional, concise, action-oriented? | 0-1 numeric |
| **Hallucination Check** | Did the agent invent services, prices, or hours not in the prompt? | boolean |
| **Business Hours Compliance** | Did the agent reject out-of-hours requests without calling availability tools? | boolean |

These run asynchronously and don't add latency to calls. Results appear in Langfuse dashboards.

### Offline Experiments (Pre-Deploy Validation)

Convert our manual test scenarios into Langfuse datasets:

**Dataset: "Integration Test Scenarios"**
- Each dataset item = one scenario from `voice-agent-integration-test-scenarios.md`
- Input: `{ "integration": "google-calendar", "caller_type": "new", "user_messages": ["Hi, my kitchen faucet is leaking..."], "location_config": {...} }`
- Expected output: `{ "tools_called": ["validate-address", "submit-intake-answers", "google-calendar-check-availability", "google-calendar-create-event"], "must_collect": ["name", "address", "email"] }`

Run experiments via SDK when prompt changes are made — compare scores across prompt versions.

## Part 3: LiveKit pytest Testing

### Why this is the primary CI tool

LiveKit's built-in testing framework is **purpose-built for our exact use case**:

- **Text-only mode** — no STT/TTS costs, just LLM calls
- **No LiveKit API keys needed** — runs purely locally
- **`judge()` method** — LLM-as-a-Judge built into assertions
- **Tool call assertions** — verify exact tool name + arguments
- **Mock tools** — simulate Jobber/HCP/GCal responses without real API calls
- **Multi-turn conversations** — test full intake → booking flows
- **Works with pytest** — integrates with CI (GitHub Actions)

### Test Structure

```
livekit-python/
├── tests/
│   ├── conftest.py              # Shared fixtures (LLM, session factory)
│   ├── test_no_integration.py   # NONE-NC-1 through NONE-EC-7
│   ├── test_google_calendar.py  # GCAL-NC-1 through GCAL-EC-*
│   ├── test_jobber.py           # JOB-NC-1 through JOB-EC-*
│   ├── test_housecall_pro.py    # HCP-NC-1 through HCP-EC-*
│   ├── test_safety.py           # Cross-integration safety scenarios
│   ├── test_returning_callers.py # Cross-integration returning caller flows
│   └── mocks/
│       ├── jobber_responses.py  # Mock Jobber API responses
│       ├── hcp_responses.py     # Mock HCP API responses
│       └── gcal_responses.py    # Mock GCal API responses
```

### Example: Mapping a Manual Scenario to pytest

**Manual scenario (GCAL-NC-1: Full booking happy path):**
> Call from unknown number → intake flow → validate address → submit intake → check availability → create event → confirm

**Automated pytest equivalent:**

```python
@pytest.mark.asyncio
async def test_gcal_new_caller_full_booking():
    """GCAL-NC-1: Full booking happy path."""
    async with (
        openai.LLM(model="gpt-4o-mini") as llm,
        AgentSession(llm=llm) as session,
    ):
        agent = create_test_agent(
            integration="google-calendar",
            location_config=GCAL_LOCATION_CONFIG,
        )

        with mock_tools(agent.__class__, {
            "validate_address": lambda address: '{"validated": "742 Evergreen Terrace, Santa Cruz, CA 95060", "city": "Santa Cruz"}',
            "submit_intake_answers": lambda **kwargs: '{"success": true}',
            "google_calendar_check_availability": lambda **kwargs: '{"available_slots": [{"start": "2026-02-27T09:00", "end": "2026-02-27T17:00"}]}',
            "google_calendar_create_event": lambda **kwargs: '{"event_id": "evt_123", "start": "2026-02-27T14:00"}',
        }):
            # Turn 1: Opening
            r1 = await session.run(user_input="Hi, my kitchen faucet is leaking really badly")
            await r1.expect.next_event(type="message").judge(
                llm, intent="Asks for the caller's name"
            )

            # Turn 2: Name
            r2 = await session.run(user_input="Alex Sikand, A-L-E-X S-I-K-A-N-D")
            await r2.expect.next_event(type="message").judge(
                llm, intent="Asks for the caller's address"
            )

            # Turn 3: Address
            r3 = await session.run(user_input="742 Evergreen Terrace, Santa Cruz, California 95060")
            r3.expect.contains_function_call(name="validate_address")

            # Turn 4: Confirm address + ask for preferred time
            r4 = await session.run(user_input="Yes that's correct")
            r4.expect.contains_function_call(name="submit_intake_answers")

            # Turn 5: Schedule
            r5 = await session.run(user_input="How about Friday at 2 PM?")
            r5.expect.contains_function_call(name="google_calendar_check_availability")
            r5.expect.contains_function_call(name="google_calendar_create_event")

            await r5.expect.next_event(type="message").judge(
                llm, intent="Confirms the appointment date, time, and address"
            )
```

## Part 4: Implementation Roadmap

### Phase 1: Foundation (Week 1) — 3-4 days

- [ ] **Set up pytest infrastructure** — `tests/conftest.py` with fixtures for creating test agents with different integration configs
- [ ] **Create mock tool responses** — `tests/mocks/` with realistic responses for each integration
- [ ] **Write P0 tests** — Core happy paths for all 4 integration modes (~15 tests)
- [ ] **Set up Langfuse online evaluators** — Tool Usage Correctness + Intake Completeness on staging traces
- [ ] **Create first Langfuse dataset** — Import P0 scenarios as dataset items

### Phase 2: Coverage (Week 2) — 3-4 days

- [ ] **Write P1 tests** — Returning callers, scheduling edge cases, config toggles (~25 tests)
- [ ] **Add Langfuse evaluators** — Safety Compliance, Tone, Hallucination Check
- [ ] **CI integration** — GitHub Actions workflow running pytest on PRs
- [ ] **Build Langfuse dashboard** — Quality score trends, tool usage rates, latency

### Phase 3: Continuous Improvement (Week 3+) — ongoing

- [ ] **Write P2 tests** — Rare edge cases (~10 tests)
- [ ] **Langfuse → Dataset loop** — Script to export flagged production traces into test datasets
- [ ] **Annotation queue** — Set up human review workflow for low-score calls
- [ ] **Experiment workflow** — Run Langfuse experiments before deploying prompt changes

### CI Pipeline

```
PR opened
  → pytest tests/test_*.py (text-only, ~2-3 min)
    → All P0 pass? → Merge allowed
    → Any P0 fail? → Block merge
    → P1/P2 fail? → Warning annotation on PR

Deploy to staging
  → Langfuse traces auto-scored by online evaluators
  → Dashboard shows quality metrics
  → Low-score calls routed to annotation queue

Deploy to production
  → Same online evaluation as staging
  → Environment filter in Langfuse separates staging/production data
```

## Summary: What Goes Where

| What | Where | Why |
|------|-------|-----|
| System prompt generation | TypeScript (`src/utils.ts`) | Too dynamic for Langfuse prompt management |
| Pre-deploy behavioral tests | LiveKit pytest (CI) | Text-only, fast, mockable, blocks bad deploys |
| Trace visibility | Langfuse tracing (already set up) | See every LLM/STT/TTS/tool call per session |
| Auto quality scoring | Langfuse online LLM-as-a-Judge | Score every production call automatically |
| Prompt change validation | Langfuse experiments via SDK | Compare before/after on fixed dataset |
| Edge case capture | Langfuse → dataset items | Real traces become regression tests |
| Human review | Langfuse annotation queues | Review flagged/low-score calls |
| End-to-end audio testing | Third-party (Hamming/Bluejay) | Future — full SIP + audio pipeline testing |
